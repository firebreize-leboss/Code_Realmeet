# Fix — Vérifier unicité email et téléphone à l'inscription

## Contexte

Dans le wizard d'inscription (`app/auth/signup-wizard.tsx`), à l'étape 3 (`components/signup/steps/Step3Contact.tsx`), l'utilisateur saisit son email et son numéro de téléphone. Actuellement, si l'email ou le numéro est déjà utilisé par un autre compte, l'erreur n'apparaît qu'à la fin de l'inscription. Il faut vérifier l'unicité **plus tôt** :

- **Email** : vérifier quand l'utilisateur clique "Continuer" à l'étape 3 (ou quand il quitte le champ email)
- **Téléphone** : vérifier quand l'utilisateur clique "Vérifier mon numéro", **avant** d'envoyer le SMS OTP (pour ne pas gaspiller un crédit MessageBird sur un numéro déjà pris)

## Ce qu'il faut implémenter

### 1. Créer une RPC Supabase pour vérifier l'unicité

Créer une migration SQL avec une fonction RPC `check_contact_availability` :

```sql
CREATE OR REPLACE FUNCTION public.check_contact_availability(
  p_email text DEFAULT NULL,
  p_phone text DEFAULT NULL
)
RETURNS json
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  result json;
  email_taken boolean := false;
  phone_taken boolean := false;
BEGIN
  -- Vérifier l'email dans auth.users
  IF p_email IS NOT NULL AND p_email != '' THEN
    SELECT EXISTS(
      SELECT 1 FROM auth.users WHERE lower(email) = lower(p_email)
    ) INTO email_taken;
  END IF;

  -- Vérifier le téléphone dans profiles (format international, ex: +33767989253)
  IF p_phone IS NOT NULL AND p_phone != '' THEN
    SELECT EXISTS(
      SELECT 1 FROM public.profiles WHERE phone = p_phone
    ) INTO phone_taken;
  END IF;

  result := json_build_object(
    'email_taken', email_taken,
    'phone_taken', phone_taken
  );

  RETURN result;
END;
$$;
```

Cette fonction est `SECURITY DEFINER` car elle accède à `auth.users` qui n'est pas accessible aux utilisateurs anonymes.

### 2. Modifier Step3Contact.tsx

Dans `components/signup/steps/Step3Contact.tsx`, ajouter deux vérifications :

**A) Vérification du téléphone — dans `handleSendOtp` (avant l'envoi OTP) :**

Avant d'appeler `phoneVerificationService.sendOtp()`, vérifier que le numéro n'est pas déjà utilisé :

```typescript
const handleSendOtp = async () => {
  if (!phoneIsValid || cooldown > 0) return;

  setOtpStatus('sending');
  setOtpError(null);

  // Vérifier que le numéro n'est pas déjà utilisé
  const { data, error } = await supabase.rpc('check_contact_availability', {
    p_phone: fullPhone,
  });

  if (error) {
    setOtpStatus('error');
    setOtpError('Erreur de vérification, réessayez');
    return;
  }

  if (data?.phone_taken) {
    setOtpStatus('error');
    setOtpError('Ce numéro de téléphone est déjà associé à un compte');
    return;
  }

  // Si le numéro est dispo, envoyer l'OTP
  const result = await phoneVerificationService.sendOtp(fullPhone);
  // ... suite existante
};
```

**B) Vérification de l'email — quand l'utilisateur perd le focus sur le champ email OU quand il clique "Vérifier mon numéro" :**

Ajouter un state `emailError` local et une fonction de vérification :

```typescript
const [emailTakenError, setEmailTakenError] = useState<string | null>(null);

const checkEmailAvailability = async () => {
  if (!formData.email || !emailRegex.test(formData.email)) return;
  
  const { data, error } = await supabase.rpc('check_contact_availability', {
    p_email: formData.email,
  });

  if (!error && data?.email_taken) {
    setEmailTakenError('Cette adresse email est déjà utilisée');
  } else {
    setEmailTakenError(null);
  }
};
```

- Appeler `checkEmailAvailability()` dans le `onBlur` du champ email
- Afficher `emailTakenError` sous le champ email (en plus de l'erreur de format existante `errors.email`)
- Si `emailTakenError` est défini, désactiver le bouton "Vérifier mon numéro" et le bouton "Continuer"

### 3. Importer supabase dans Step3Contact

Ajouter l'import :
```typescript
import { supabase } from '@/lib/supabase';
```

### 4. Mettre à jour la validation dans SignupContext (optionnel mais recommandé)

Dans `contexts/SignupContext.tsx`, la validation de l'étape 3 vérifie déjà le format. La vérification d'unicité est faite côté composant (asynchrone), donc elle ne peut pas être dans `getStepErrors` (qui est synchrone). C'est correct de la garder dans le composant.

## Fichiers à créer
- `supabase/migrations/YYYYMMDD_check_contact_availability.sql` — la fonction RPC

## Fichiers à modifier
- `components/signup/steps/Step3Contact.tsx` — ajouter la vérification email (onBlur) + vérification téléphone (dans handleSendOtp avant l'envoi OTP)

## Comportement attendu

1. L'utilisateur saisit un email déjà utilisé → quand il quitte le champ, le message "Cette adresse email est déjà utilisée" apparaît en rouge sous le champ
2. L'utilisateur saisit un numéro déjà utilisé → quand il clique "Vérifier mon numéro", le message "Ce numéro de téléphone est déjà associé à un compte" apparaît, et **aucun SMS n'est envoyé**
3. Si l'email et le numéro sont disponibles, le flow continue normalement (envoi OTP, etc.)
4. Les messages d'erreur disparaissent quand l'utilisateur modifie le champ concerné

## Migration SQL à appliquer

Après avoir créé le fichier de migration, rappeler à l'utilisateur d'exécuter la fonction SQL dans le Dashboard Supabase → SQL Editor.

Dans app/activity-detail.tsx, dans la fonction loadActivity, il y a un bug : la boucle qui cherche une participation active sur un slot futur ne vérifie pas si le slot est annulé (is_cancelled). Du coup si l'utilisateur a une participation active sur un slot annulé ET un slot non-annulé, le code peut tomber sur le slot annulé en premier, faire break, puis reset isJoined à false sans jamais vérifier le slot valide.

Fix : dans la requête Supabase qui récupère les participations actives, ajoute un JOIN avec activity_slots pour exclure les slots annulés dès la requête. Remplace :

const { data: participations } = await supabase
    .from('slot_participants')
    .select('id, slot_id')
    .eq('activity_id', activityId)
    .eq('user_id', userId)
    .eq('status', 'active');

Par :

const { data: participations } = await supabase
    .from('slot_participants')
    .select('id, slot_id, activity_slots!inner(date, time, is_cancelled)')
    .eq('activity_id', activityId)
    .eq('user_id', userId)
    .eq('status', 'active')
    .eq('activity_slots.is_cancelled', false);

Et adapte la boucle en dessous pour utiliser activity_slots.date et activity_slots.time depuis le JOIN au lieu de refaire une requête séparée pour chaque slot.

# Prompt Claude Code — Vérification SMS MessageBird dans l'inscription RealMeet

## Contexte

L'app RealMeet est une app React Native (Expo) avec Supabase. L'inscription se fait via un wizard multi-étapes (`app/auth/signup-wizard.tsx`) piloté par `contexts/SignupContext.tsx`. Le téléphone est collecté à l'étape 3 (`components/signup/steps/Step3Contact.tsx`). Le composant `components/PhoneInput.tsx` gère déjà la saisie et validation du numéro.

**Objectif** : Ajouter une étape de vérification OTP par SMS via la configuration Supabase Phone (provider MessageBird) APRÈS la saisie du téléphone à l'étape 3, et AVANT de finaliser l'inscription. Le numéro doit être vérifié pour empêcher les faux comptes.

## Architecture existante à connaître

- **Wizard** : 7 étapes → review → submit → success
- **SignupContext** (`contexts/SignupContext.tsx`) : gère `formData`, `currentStep`, `TOTAL_STEPS = 7`, validation par étape
- **Step3Contact** (`components/signup/steps/Step3Contact.tsx`) : collecte email + téléphone
- **PhoneInput** (`components/PhoneInput.tsx`) : composant de saisie téléphone avec `PhoneValue`, `validatePhone()`, `formatFullPhone()`
- **AuthService** (`services/auth.service.ts`) : `registerUser()` crée le compte Supabase + profil
- **signup-wizard.tsx** : orchestre les étapes, `handleSubmit()` fait l'inscription finale
- **Supabase config** (`supabase/config.toml`) : SMS configuré avec section `[auth.sms]` et provider `[auth.sms.twilio]` (à changer pour messagebird)
- **Numéro de test configuré dans Supabase Dashboard** : `33767989253=000000`

## Ce qu'il faut implémenter

### 1. Configurer Supabase pour MessageBird

Dans le **Dashboard Supabase** (pas dans le code, juste documenter les étapes) :
- Aller dans Authentication → Providers → Phone
- Activer le provider Phone
- Sélectionner MessageBird comme provider SMS
- Renseigner l'Access Key MessageBird
- Mettre `RealMeet` comme Originator
- Ajouter le numéro de test : `33767989253` avec OTP `000000`

Dans `supabase/config.toml` pour le dev local, ajouter/modifier :
```toml
[auth.sms]
enable_signup = false          # On ne veut PAS le signup par SMS, juste la vérification
enable_confirmations = false
template = "Votre code RealMeet : {{ .Code }}"
max_frequency = "30s"

[auth.sms.test_otp]
33767989253 = "000000"

[auth.sms.messagebird]
enabled = true
originator = "RealMeet"
access_key = "env(SUPABASE_AUTH_SMS_MESSAGEBIRD_ACCESS_KEY)"
```

### 3. Modifier le SignupContext

Dans `contexts/SignupContext.tsx` :

- Ajouter dans `SignupFormData` :
  ```typescript
  phoneVerified: boolean;    // Le numéro a-t-il été vérifié par OTP ?
  ```
- Initialiser `phoneVerified: false` dans `initialFormData`
- **NE PAS** changer `TOTAL_STEPS` (reste 7) — la vérification OTP est une sous-étape de l'étape 3
- Ajouter une validation : à l'étape 3, si `phoneVerified === false`, l'étape n'est pas valide (le bouton "Continuer" reste grisé tant que le téléphone n'est pas vérifié)
- Si l'utilisateur modifie son numéro après l'avoir vérifié, remettre `phoneVerified = false`

### 4. Modifier Step3Contact pour intégrer la vérification OTP

Dans `components/signup/steps/Step3Contact.tsx`, ajouter un flow en 2 temps :

**État initial** : L'utilisateur saisit son email et son téléphone normalement.

**Quand le numéro est valide** : Afficher un bouton "Vérifier mon numéro" sous le champ téléphone.

**Au clic sur "Vérifier"** :
1. Appeler `phoneVerificationService.sendOtp(formattedPhone)`
2. Afficher un champ de saisie OTP (6 chiffres) avec un design soigné
3. Afficher un timer de renvoi (30s minimum entre chaque envoi)
4. Un bouton "Renvoyer le code" grisé pendant le cooldown

**Quand l'utilisateur saisit le code** :
1. Appeler `phoneVerificationService.verifyOtp(phone, code)`
2. Si succès : `updateFormData('phoneVerified', true)`, afficher un check vert, masquer le champ OTP
3. Si échec : afficher l'erreur, permettre de réessayer

**UI du champ OTP** :
- 6 cases individuelles (comme les apps bancaires)
- Auto-focus sur la case suivante à chaque chiffre
- Style cohérent avec le design existant (utiliser `colors`, `spacing`, `typography` de `commonStyles`)
- Animation subtile au succès (check vert qui apparaît)

**Si l'utilisateur change son numéro après vérification** :
- Remettre `phoneVerified = false`
- Masquer le check vert
- Redemander la vérification

### 5. Modifier la validation dans le wizard

Dans `signup-wizard.tsx`, dans `handleSubmit()` :
- Vérifier que `formData.phoneVerified === true` avant de soumettre
- Si non vérifié (ne devrait pas arriver grâce à la validation d'étape), afficher une alerte

### 6. Stocker le statut vérifié dans le profil

Quand l'inscription est finalisée (`handleSubmit` dans `signup-wizard.tsx`) :
- Le numéro est déjà envoyé au profil via `authService.registerUser()`
- Ajouter un champ `phone_verified: true` dans la table `profiles` (ou utiliser le champ existant si disponible)
- S'assurer que le numéro est stocké au format international (`+33767989253`)

### 7. Considérations importantes

- **Ne PAS casser le flow existant** : le wizard doit continuer à fonctionner normalement
- **Ne PAS créer de nouveau screen/page** : la vérification se fait inline dans Step3Contact
- **Gérer le cas hors-ligne** : si pas de réseau, afficher un message d'erreur clair
- **Rate limiting** : respecter le `max_frequency` de 30s entre chaque envoi
- **Numéro de test** : pour le dev, le numéro `+33767989253` avec le code `000000` doit fonctionner via les test OTP configurés dans Supabase
- **Nettoyage de session** : si `signInWithOtp` + `verifyOtp` crée une session Supabase temporaire, la détruire (`supabase.auth.signOut()`) car l'utilisateur n'est pas encore inscrit. L'inscription réelle se fait après via `registerUser()` avec email/password.
